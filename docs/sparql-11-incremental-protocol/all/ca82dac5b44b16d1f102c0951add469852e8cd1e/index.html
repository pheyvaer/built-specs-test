Abstract

This document defines an extension to the SPARQL 1.1 Protocol that enables incremental delivery of SPARQL query results over Server-Sent Events (SSE). Instead of a single response, the server emits a stream of events that convey an initial complete result set followed by incremental updates and explicit "up-to-date" signals. This extension is intended for use by incremental view maintenance (IVM) engines and other systems that require low-latency maintenance of SPARQL query results under updates.

1. Introduction

The SPARQL Protocol defines how clients issue SPARQL queries and receive results over HTTP. In its standard form, each query produces a single response containing the complete result set, serialized using formats such as SPARQL Results JSON or SPARQL Results XML.

For applications that maintain long-lived queries over changing data — such as incremental view maintenance (IVM) engines, live dashboards, and stream processors — repeatedly reissuing queries and re-fetching full results is inefficient and adds latency.

This specification defines an extension to the SPARQL Protocol that allows a SPARQL Protocol service to deliver results as a server-sent events event stream. Instead of returning a single result document, the service emits:

  • an initial event containing a complete result set;

  • zero or more update events conveying incremental additions and deletions; and

  • one or more up-to-date events indicating that, at the moment of emission, the service has processed all known changes for the query.

This extension is designed to:

  • integrate cleanly with the existing SPARQL 1.1 Protocol;

  • be compatible with the EventSource API in JavaScript;

  • support both stateless query services and stateful IVM engines; and

  • allow (optional) checksums for end-to-end consistency checking.

This version of the specification normatively defines JSON-based SSE result streams. Other serializations, such as XML-based SSE result streams, are out of scope for this version and may be defined by future specifications.

2. Conformance

This specification defines the following classes of products:

  • SSE IVM SPARQL services (or simply “services”): HTTP services that implement the SPARQL Protocol and the SSE extensions defined in this specification.

  • SSE IVM SPARQL clients (or simply “clients”): HTTP clients that request and consume SSE result streams as defined in this specification.

The key words “MUST”, “MUST NOT”, “REQUIRED”, “SHOULD”, “SHOULD NOT”, and “MAY” in this document are to be interpreted as described in BCP 14 (RFC 2119 and RFC 8174) when, and only when, they appear in all capitals.

3. Terminology

This specification uses the terminology of [SPARQL11-QUERY], [SPARQL11-PROTOCOL], and [SPARQL11-RESULTS-JSON]. In particular:

  • A solution sequence is as defined by [SPARQL11-QUERY].

  • SPARQL query, SPARQL Protocol service, and related HTTP concepts are as defined by [SPARQL11-PROTOCOL].

  • SPARQL Results JSON is as defined by [SPARQL11-RESULTS-JSON].

In addition, this document uses the following terms:

SSE result stream
A server-sent events event stream whose events encode SPARQL query results according to this specification.
Initial event
An SSE event named initial that contains a complete SPARQL result set for a query.
Update event
An SSE event named update that contains incremental additions and deletions to the query’s solution sequence, relative to previously delivered events.
Up-to-date event
An SSE event named up-to-date that indicates that, at the time of emission, the service has processed all known changes affecting the query result.
Checksum
An opaque string value that identifies a particular state of a query result set. Equality of checksums indicates that the corresponding result sets are equal. The checksum algorithm is intentionally unspecified in this version.

4. Media Types and Negotiation

SSE result streams are carried over HTTP using the text/event-stream media type defined by [HTML]. For compatibility with the JavaScript EventSource API, SSE IVM SPARQL services MUST set:

  • Content-Type: text/event-stream; charset=utf-8

  • Cache-Control: no-cache

  • Connection: keep-alive (for HTTP/1.1)

4.1. Selecting SSE vs normal SPARQL results

Clients select SSE delivery using HTTP content negotiation:

  • If the Accept header includes text/event-stream, the service MAY return an SSE result stream.

  • If the service does not support SSE, it MUST ignore text/event-stream and return a normal SPARQL results document using the rules of [SPARQL11-PROTOCOL].

The Content-Type of an SSE response is always exactly:

text/event-stream

No additional suffixes or variants (such as text/event-stream+sparql-results+json) are valid or permitted.

4.2. Selecting the internal SPARQL results serialization

Because SSE only defines one legal outer media type, the serialization used inside SSE data: payloads is negotiated using a query parameter:

serialization = json | xml

Defined as follows:

  • If serialization=json, the service MUST encode event payloads in JSON.

  • If serialization=xml, the service MUST encode event payloads in SPARQL Results XML.

  • If serialization is omitted, the default is json.

If a client specifies a value that the server does not support:

  • The server SHOULD return 406 Not Acceptable.

The serialization parameter applies equally to GET and POST SPARQL queries.

5. SSE Result Stream Binding

This section defines how a SPARQL Protocol service delivers query results over a Server-Sent Events (SSE) result stream. The stream consists of the following event types (defined in detail below):

  • initial — a complete result set for the query (see § Initial event).

  • update — incremental additions and deletions relative to the previous state (see § Update event).

  • up-to-date — a signal that all known committed changes have been reflected (see § Up-to-date event).

5.1. Serialization Parameter

An SSE IVM SPARQL service MUST support a serialization query parameter on SPARQL GET requests and on the request-body-encoded parameters of SPARQL POST requests.

The parameter is defined as:

serialization = json | xml

Semantics:

  • Determines the syntax used inside SSE event payloads.

  • Does not affect the HTTP Content-Type, which remains text/event-stream.

  • Applies uniformly to the payloads of all events (initial, update, and up-to-date).

5.2. GET example

GET /sparql?query=SELECT+...&serialization=xml
Accept: text/event-stream

5.3. POST example (URL-encoded)

POST /sparql
Content-Type: application/x-www-form-urlencoded
Accept: text/event-stream

query=SELECT+...&serialization=json

If omitted, the default value is:

serialization=json

5.4. Overview

An SSE IVM SPARQL service extends the SPARQL Protocol as follows:

  • The query is issued using the existing HTTP methods and parameters defined in [SPARQL11-PROTOCOL] (e.g., HTTP GET with query parameter or HTTP POST with body).

  • The client uses the Accept header to request an SSE result stream, as described in § Media Types and Negotiation.

  • If the service elects to deliver results as an SSE result stream, it responds with:

  • HTTP status 200 OK;

  • headers including Content-Type: text/event-stream; and

  • a body that is a server-sent events event stream.

Within that stream:

  • Exactly one initial event MUST be sent, conveying a complete result set.

  • Zero or more update events MAY be sent, each conveying incremental additions and deletions.

  • One or more up-to-date events MUST be sent at moments when the result is known to be current with respect to all changes processed so far.

The stream continues until either:

  • the client closes the connection; or

  • the service unilaterally terminates the stream (for example, due to resource limits or administrative reasons).

Clients MUST be prepared for the stream to terminate at any time, and MUST handle partial result histories gracefully.

5.5. Initial event

For SSE streams, the payload of an initial event MUST be a complete SPARQL result set encoded using the format selected by the serialization parameter. The default is SPARQL Results JSON.

Example (abbreviated):

event: initial data: { data: "head": { "vars": [ "x", "y" ] }, data: "results": { data: "bindings": [ data: { "x": { "type": "uri", "value": "http://example.org/a" }, data: "y": { "type": "literal", "value": "1", "datatype": "http://www.w3.org/2001/XMLSchema#integer" } } data: ] data: } data: }

The initial event MAY include a top-level "checksum" field in addition to the fields defined by [SPARQL11-RESULTS-JSON]. When present, this field MUST be a JSON string and MUST conform to the checksum semantics described in the Checksums section.

5.6. Update event

For SSE result streams, the payload of an update event MUST be encoded using the serialization specified by the serialization parameter. This specification normatively defines the JSON encoding. If serialization=xml is used, an implementation-defined XML encoding of additions and deletions MAY be used.

{ "additions": { "bindings": [ /* array of binding objects, as in SPARQL Results JSON / ] }, "deletions": { "bindings": [ / array of binding objects, as in SPARQL Results JSON / ] }, "checksum": "..." / optional */ }

Where:

"additions"

REQUIRED. An object with a "bindings" array.

Each entry in "bindings" MUST be a binding object as defined for results.bindings in [SPARQL11-RESULTS-JSON].

The bindings in additions.bindings represent solution mappings newly added to the result set, relative to the most recent prior state.

If there are no additions, additions.bindings MAY be an empty array.

"deletions"

REQUIRED. An object with a "bindings" array.

Each entry in deletions.bindings MUST be a binding object as defined for results.bindings in [SPARQL11-RESULTS-JSON].

The bindings in deletions.bindings represent solution mappings removed from the result set, relative to the most recent prior state.

If there are no deletions, deletions.bindings MAY be an empty array.

"checksum"

OPTIONAL. A JSON string conforming to the checksum semantics described later in this document (see the Checksums section).

Services MUST apply deletions and additions atomically for the logical state represented by the update event. Within the event, deletions conceptually occur before additions when computing the new result set.

Clients that maintain a local view of the result set should remove all bindings specified in deletions.bindings and then add all bindings specified in additions.bindings.

5.7. Up-to-date event

The payload of an up-to-date event MUST be encoded using the serialization selected by the serialization parameter. In JSON mode, the payload is an object containing an optional "checksum" string. In XML mode, the payload is an XML element conveying the same information.

{ "checksum": "..." /* optional */ }

Where:

"checksum"
OPTIONAL. A string conforming to the semantics described later in the Checksums section.

An up-to-date event MUST NOT introduce additions or deletions. It is purely a signal of the service’s maintenance state. Services SHOULD emit an up-to-date event after the initial event (once the initial result set has been fully produced and no further changes are pending) and after applying each subsequent batch of changes, whenever the result has again become current with respect to all known committed changes.

5.8. Event Structure

Each SSE event has the form defined in [HTML]:

event: <event-name>
data: <payload-line-1>
data: <payload-line-2>
...

The payload of an event is obtained by concatenating the data: lines with \n (U+000A) line separators, as defined by [HTML]. For JSON-based streams (the only normative serialization in this version):

  • The payload of each event MUST be a single well-formed JSON text.

  • JSON documents MUST be encoded in UTF-8.

5.8.1. Event types {#event-types}

The following event types are defined by this specification:

initial
Conveys a complete SPARQL result set for the query. Exactly one initial event MUST appear in a stream, and it MUST be the first event carrying query results.
update
Conveys incremental additions and deletions relative to the most recently delivered logical result state. Zero or more update events MAY appear.
up-to-date
Signals that, at the time of emission, all known committed changes affecting the query have been reflected in the result state. One or more up-to-date events MUST appear (at least once after the initial event).

User agents and clients MUST ignore any SSE events whose event: value is not one of the above, unless otherwise negotiated out-of-band.

Event Ordering and Semantics {#event-order}

Within an SSE result stream:

A service MUST emit exactly one initial event as the first event that carries query results.

After the initial event, the service MAY emit any number of update and up-to-date events.

For any given query:

The sequence of initial and update events collectively describes a sequence of result sets.

Applying the additions and deletions in order yields a sequence of solution sequences, each of which MUST be a valid result of the query over some logical state of the underlying data.

An up-to-date event indicates that, at the time it is sent, all changes to the underlying data that the service has accepted and committed up to that moment have been reflected in the result set.

Services MAY batch changes arbitrarily:

A single update event MAY contain multiple additions and deletions.

A service MAY emit update events per binding, per group of bindings, or per internal update cycle.

If there are no changes in a particular maintenance cycle, the service MAY omit an update event entirely.

Clients MUST NOT assume any fixed mapping between underlying data operations and individual update events.

Checksums {#checksums}

Checksums are OPTIONAL and are intentionally specified as opaque strings. Checksum semantics {#checksum-semantics}

A checksum is any value chosen by the service such that:

For a given query and set of result bindings, the service can compute a checksum string.

If two checksums produced by the same service, for the same query and query parameters, are equal, then the corresponding result sets MUST be equal as multisets of solution mappings.

No particular checksum algorithm is mandated by this specification. Implementations MAY use cryptographic hashes, non-cryptographic hashes, version identifiers, or any other mechanism that meets the above criteria. Checksum placement {#checksum-placement}

Checksums MAY appear in:

the initial event payload, as a top-level "checksum" field;

any update event payload, as a top-level "checksum" field; and

any up-to-date event payload, as a top-level "checksum" field.

Checksums MAY be omitted from any or all events.

Clients that understand checksums MAY use them to:

verify that the locally maintained result set matches the service’s view; and

detect and recover from missed update events (for example, after reconnecting).

Behavior Under Updates {#update-behavior} Update model {#update-model}

This specification is agnostic to how the service maintains query results under data updates (e.g., via incremental view maintenance, recomputation, or hybrid strategies). However:

Every sequence of initial and update events MUST correspond to some valid sequence of result sets of the query over a sequence of logical database states.

Services MUST ensure that deletions are only announced for solution mappings that have previously been present in the result set (either in the initial event or in some earlier update event).

Granularity and batching {#granularity}

Services have full freedom in choosing the granularity of update events:

An update event MAY represent a single logical change (e.g., one inserted triple).

An update event MAY represent multiple logical changes, consolidated into a single batch of additions and deletions.

Services MAY coalesce changes to the same solution mapping into a single event, or split them across multiple events.

To reflect the user requirement:

Updates don’t need to be sent when all changes have been processed. This can be binding per binding.

This specification clarifies that:

A service is not required to emit an update event if no changes occurred during a maintenance cycle.

A service MAY emit update events either per binding (fine-grained) or per batch of bindings (coarse-grained).

Stream Lifecycle {#lifecycle} Stream start {#stream-start}

Upon receiving a SPARQL query request whose Accept header indicates that the client can accept an SSE result stream:

The service determines whether it is willing and able to serve this query via SSE.

If so, it responds with HTTP 200 OK, sets Content-Type: text/event-stream, and begins emitting SSE events as specified above.

If not, it responds as a regular SPARQL Protocol service, ignoring the SSE-related Accept values.

Stream termination {#stream-termination}

An SSE result stream terminates when:

the client closes the connection; or

the service closes the connection.

Services MAY terminate an SSE stream at any time, for example:

due to administrative policy;

due to resource exhaustion;

after a configured maximum duration; or

when the query result becomes known to be immutable (e.g., against a static dataset), after having emitted an up-to-date event.

Upon termination, no further guarantees are made about the completeness of any subsequent local maintenance performed by the client. Clients SHOULD treat stream termination as an indication that the server will not send further updates and SHOULD fall back to other mechanisms if continued maintenance is required (e.g., by issuing a new query). Error Handling {#errors}

Transport-level errors (e.g., network interruptions) are handled as defined by HTTP and [HTML] for server-sent events. This specification does not introduce additional error signaling within the SSE protocol.

Services MAY choose to encode application-level errors as SSE events with other event names, but such events are not defined by this version of the specification. Clients MUST ignore SSE events whose event name is not initial, update, or up-to-date, unless they have an out-of-band agreement to process them. Security Considerations {#security}

This specification introduces a long-lived HTTP connection per active query. Implementers SHOULD consider:

Resource limits to prevent a single client from exhausting server resources with many concurrent SSE queries.

Authentication and authorization for queries and for continued subscription to updates.

Confidentiality and integrity of streams (e.g., by using HTTPS).

Checksums, if implemented with cryptographic hashes, SHOULD be chosen to avoid leaking sensitive information about the underlying data beyond what is already exposed by the query results themselves. Privacy Considerations {#privacy}

Incremental result delivery can make the timing and pattern of updates observable to clients, which may reveal information about the underlying system’s update behavior. Services and deployments concerned with such side channels SHOULD consider rate-limiting, batching, or other mitigations.
